# AGENTS.md

## Обзор frontend-части

Директория `web/` содержит клиентскую часть упрощённого приложения бронирования встреч, вдохновлённого Cal.com.

Frontend должен покрывать два основных сценария:

1. **Public booking flow**

- гость просматривает публичные типы событий
- открывает конкретный тип события по `slug`
- видит доступные слоты на ближайшие 14 дней
- создаёт бронирование без аутентификации

1. **Admin panel**

- владелец управляет типами событий
- владелец управляет weekly availability
- владелец управляет availability overrides
- владелец просматривает предстоящие бронирования

Во frontend-части нет аутентификации, регистрации и пользовательских ролей в терминах security. Админская часть существует как отдельный набор маршрутов и экранов для одного предопределённого владельца.

## Цель frontend-части

Клиент должен:

- прозрачно работать с API, описанным в `../typespec/`
- не дублировать контракт вручную
- отображать локальное время пользователю, не ломая UTC-контракты backend
- разделять public и admin сценарии на уровне маршрутов, layout и feature-модулей
- оставаться простым, предсказуемым и пригодным для расширения

## Технологический стек

Frontend строится на следующих технологиях:

- `Next.js` с `App Router`
- `React`
- `TypeScript`
- `Mantine` как основная UI-библиотека
- `next-intl` для i18n и хранения текстов интерфейса по локалям
- `@tanstack/react-query` для server state
- `openapi-typescript` для генерации типов из OpenAPI
- `openapi-fetch` как typed API client
- `Prism` для mock и contract validation
- `Luxon` для форматирования и отображения дат и времени
- `Vitest`
- `Testing Library`
- `MSW`

## Архитектурные принципы frontend

### 1. Контракт приходит из TypeSpec

`../typespec/` остаётся единственным источником истины для API.

Это означает:

- frontend не объявляет вручную дублирующие DTO для request и response
- типы клиента генерируются из `../typespec/tsp-output/openapi.json`
- любое изменение API начинается в `TypeSpec`, а не в `web/`

### 2. Frontend не содержит бизнес-логику бронирования

Frontend не должен решать доменные задачи backend:

- не вычисляет доступные слоты самостоятельно
- не определяет длительность бронирования как источник истины
- не реализует проверку пересечений
- не реализует правила booking window как доменный источник истины

Frontend только:

- отображает серверные данные
- собирает корректный пользовательский ввод
- валидирует форму на уровне UX
- показывает ошибки API и состояния загрузки

### 3. Time handling остаётся согласованным с backend

Во frontend действуют следующие правила:

- все значения `utcDateTime`, приходящие из API, считаются источником истины
- в запросах на backend отправляются UTC ISO строки, соответствующие контракту
- локальное время используется только для отображения пользователю
- recurring weekly availability не превращается во frontend-бизнес-логику расчёта слотов

### 4. App Router управляет маршрутами

Маршрутизация строится через `app/`.

`App Router` используется для:

- разделения public и admin частей приложения
- layout-композиции
- route-level loading и error states
- page-level composition

### 5. Тексты интерфейса живут в i18n-слое

Во frontend все пользовательские тексты должны храниться централизованно через `next-intl`.

Это означает:

- не хранить UI-строки россыпью по компонентам без необходимости
- использовать словари сообщений по локалям
- отделять переводимый UI-текст от доменных данных, приходящих из API
- не пытаться переводить значения, которые являются частью серверного контракта

### 6. Используется облегчённая FSD-архитектура

Проект не обязан следовать каноническому FSD во всей строгости. Для этого приложения используется облегчённый вариант:

- `app/` отвечает за маршруты и layout
- `src/features` отвечает за пользовательские сценарии
- `src/entities` отвечает за доменные сущности frontend-уровня
- `src/widgets` отвечает за крупные составные UI-блоки
- `src/shared` отвечает за общий API, UI, config и утилиты

## Правила работы с API

### 1. Генерация типов

Frontend должен использовать `openapi.json`, который генерируется из `TypeSpec`.

Ожидаемый поток:

1. обновить контракт в `../typespec/`
2. сгенерировать новый `openapi.json`
3. обновить frontend-типы из OpenAPI
4. использовать эти типы в API client и feature-коде

### 2. Typed client

Во frontend должен существовать единый API layer в `src/shared/api`.

Требования к нему:

- использовать `openapi-fetch` поверх сгенерированных типов
- инкапсулировать `baseUrl`, общую обработку ошибок и сериализацию
- разделять public и admin вызовы по модулям, а не по разрозненным `fetch` внутри страниц

### 3. Server state

Работа с данными должна строиться через `@tanstack/react-query`.

Это касается:

- списков event types
- загрузки event type по `slug`
- загрузки slots
- создания booking
- загрузки и изменения availability
- загрузки и изменения overrides
- загрузки admin bookings

## Workflow с Prism

`Prism` используется в двух режимах.

### 1. Начальный этап: mock server

Пока backend не готов, `Prism` работает как mock server поверх `openapi.json`.

Это означает:

- frontend разрабатывается против контракта, а не против вымышленных ручных моков
- public и admin сценарии можно собирать до готовности backend
- формы и состояния ошибок можно проверять на раннем этапе

### 2. Следующий этап: proxy и contract validation

Когда backend появится, `Prism` должен использоваться как прокси-слой между frontend и сервером.

Это означает:

- запросы и ответы backend проверяются на соответствие контракту
- frontend продолжает использовать тот же typed client
- `TypeSpec` остаётся источником истины, а `Prism` помогает ловить расхождения реализации

## UI и composition guidelines

При работе с `Mantine` нужно придерживаться следующих правил:

- использовать готовые компоненты библиотеки как основу интерфейса
- не тащить вторую UI-систему без явной необходимости
- не строить сложный кастомный дизайн-слой до появления реальной потребности
- сохранять единый визуальный язык между public и admin интерфейсами, но с разными layout-паттернами

Public-часть должна быть простой и понятной для гостя. Admin-часть должна быть плотной, утилитарной и ориентированной на управление данными.

## Правила маршрутов и экранов

Во frontend должны существовать как минимум следующие направления:

- `/` или аналогичный публичный стартовый экран с доступными event types
- `/book/[slug]` для публичного сценария выбора слота и создания бронирования
- `/admin/event-types`
- `/admin/availability`
- `/admin/overrides`
- `/admin/bookings`

Маршруты должны отражать реальные сценарии продукта, а не внутренние технические детали API.

## Ожидания от агентов в web/

При внесении изменений в клиентскую часть:

- сохранять contract-first подход
- не дублировать backend-типы вручную
- не переносить backend-бизнес-логику во frontend
- использовать явные названия feature и entity модулей
- предпочитать небольшие и предсказуемые компоненты
- держать маршруты в `app/`, а не смешивать их с прикладной логикой
- держать API layer централизованным
- добавлять тесты для нетривиальных сценариев

## Вне области frontend MVP

Следующее намеренно не входит в клиентский MVP:

- аутентификация и регистрация
- личные кабинеты разных пользователей
- сложная ролевая модель
- интеграции с внешними календарями
- уведомления
- платежи
- офлайн-режим
- локальная доменная логика расчёта слотов

Frontend должен оставаться простым, contract-driven и сфокусированным на двух сценариях: публичное бронирование и администрирование расписания.

Я хотел бы писать все приложение на этой библиотеки с минимальным количеством кастомных элементов, чтобы минимизировать количество кода и чтобы приложение легко поддерживалось.
Можно поменять цветовую политру библиотк и сделать ее более приближенной к макету. Все кнопки должны распологаться как на макете и выполнять свои функции
