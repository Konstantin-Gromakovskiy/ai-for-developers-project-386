# AGENTS.md

## Обзор проекта

Этот репозиторий содержит монорепозиторий для упрощённого приложения бронирования встреч, вдохновлённого Cal.com.

В системе есть две условные роли:

1. **Владелец (Owner)**
   - управляет типами событий
   - управляет доступностью (availability)
   - просматривает предстоящие бронирования

2. **Гость (Guest)**
   - просматривает публичные типы событий
   - выбирает тип события
   - видит доступные слоты на ближайшие 14 дней
   - создаёт бронирование без аутентификации

В проекте **нет аутентификации и регистрации**.  
Владелец представлен одним заранее определённым профилем, который неявно используется в админской части.

Цель проекта — создать чистое fullstack-приложение с:

- публичным процессом бронирования
- простой админ-панелью
- детерминированной логикой бронирования
- чёткими контрактами между frontend и backend

## Ключевой архитектурный принцип

### 1. TypeSpec — источник истины (Single Source of Truth)

Контракты API описываются исключительно в директории `typespec/`.

Это означает:

- все эндпоинты, DTO и модели определяются через TypeSpec
- сервер и клиент должны **генерироваться или строго соответствовать этим контрактам**
- запрещено вручную дублировать типы в `server` и `client`

Любые изменения API должны:

1. сначала вноситься в `typespec`
2. затем применяться к серверу и клиенту

### 3. Слоты — вычисляемые данные

Доступные слоты **не должны храниться в базе данных**.  
Они должны генерироваться динамически на основе:

- длительности события
- доступности владельца
- окна бронирования
- существующих бронирований

### 4. Работа со временем

Все даты и время должны храниться в едином формате, предпочтительно UTC ISO строки.  
Frontend может отображать локальное время, но логика backend должна оставаться детерминированной.

### 5. Валидация бронирования

При создании бронирования backend должен проверять:

- существует ли тип события
- слот находится в пределах 14 дней
- слот соответствует доступности владельца
- длительность соответствует выбранному типу события
- слот не пересекается с существующими бронированиями

### 6. Безопасность при конкурентности

Два гостя могут попытаться забронировать один и тот же слот одновременно.  
Создание бронирования должно быть защищено транзакциями или аналогичными гарантиями на уровне базы данных.

### 7. Чёткое разделение ответственности

Слои должны быть явно разделены:

- controllers/routes — обработка HTTP
- services — бизнес-логика
- repositories/data-access — работа с данными
- shared package — DTO, схемы и общие типы

## Ожидания от агентов

При внесении изменений:

- сохранять доменные правила
- не переносить бизнес-логику на frontend
- не дублировать типы
- писать предсказуемые функции
- использовать явные имена
- держать валидацию на границе API
- документировать сложную логику времени
- добавлять тесты

### Критические тест-кейсы

- корректная генерация слотов
- исключение занятых слотов
- отказ при пересечении
- отказ вне окна 14 дней
- корректная работа overrides
- бронирование только внутри доступности

---

## Вне области проекта

Следующее намеренно не реализуется:

- аутентификация
- регистрация
- мульти-пользовательская система
- интеграции с календарями
- уведомления
- платежи
- повторяющиеся бронирования

Приложение должно оставаться простым и сфокусированным.
