# AGENTS.md

## Обзор backend-части

Директория `server/` содержит backend-часть упрощённого приложения бронирования встреч, вдохновлённого Cal.com.

Backend должен обслуживать два сценария:

1. **Public API**
- гость получает список публичных типов событий
- гость получает детали одного типа события по `slug`
- гость получает доступные слоты на ближайшие 14 дней
- гость создаёт бронирование без аутентификации

2. **Admin API**
- владелец управляет типами событий
- владелец управляет recurring availability
- владелец управляет availability overrides
- владелец просматривает бронирования

Во backend-части нет аутентификации, регистрации и полноценной ролевой модели. Владелец один и рассматривается как неявный субъект административного API.

## Цель backend-части

Backend должен:

- строго соответствовать API-контракту из `../typespec/`
- инкапсулировать всю доменную логику бронирования
- вычислять слоты динамически, а не хранить их в базе
- безопасно создавать бронирования при конкурентных запросах
- возвращать предсказуемые ответы и ошибки
- оставаться простым, модульным и тестируемым

## Технологический стек

Backend строится на следующих технологиях:

- `NestJS`
- `TypeScript`
- `OpenAPI`, сгенерированный из `TypeSpec`
- транзакционная SQL-база данных
- репозиторный слой для доступа к данным
- `Jest` и `Nest Testing` для unit и e2e тестов

Версии библиотек в этом документе не фиксируются. При установке используются актуальные стабильные версии, если в проекте не задано иное явно.

## Архитектурные принципы backend

### 1. TypeSpec остаётся источником истины

Контракты API задаются только в `../typespec/`.

Это означает:

- backend не изобретает новые API-формы ответа вне контракта
- backend не меняет структуру DTO без изменения `TypeSpec`
- сначала меняется `TypeSpec`, затем обновляется backend-реализация
- контроллеры и response mapping должны соответствовать `openapi.json`

Если для `NestJS` понадобятся DTO-классы или runtime validation classes, они не должны превращаться во второй независимый источник типов. Контракт всё равно остаётся в `TypeSpec`.

### 2. NestJS используется как HTTP и composition слой

В `NestJS` должны быть чётко разделены обязанности:

- controllers принимают HTTP-запросы и возвращают HTTP-ответы
- services содержат доменную логику
- repositories работают с базой данных
- mappers преобразуют внутренние модели в API-контракт

Не нужно переносить бизнес-логику в:

- controllers
- decorators
- pipes
- repositories

### 3. Слоты вычисляются на лету

Доступные слоты не должны храниться в базе данных.

Они вычисляются из:

- длительности event type
- recurring availability владельца
- availability overrides
- окна в 14 дней
- уже существующих бронирований

Сервис слотов должен быть детерминированным и независимым от HTTP.

### 4. Время и даты обрабатываются централизованно

Во backend действуют следующие правила:

- все booking и slot интервалы рассматриваются как UTC
- API принимает и отдаёт UTC ISO строки, как определено в контракте
- recurring weekly availability интерпретируется относительно timezone владельца
- преобразования времени должны жить в одном общем time utility слое, а не быть размазанными по controllers и repositories

Сложная логика времени должна документироваться комментариями рядом с кодом, если без этого трудно понять поведение.

### 5. Создание бронирования должно быть конкурентно-безопасным

Backend обязан предотвращать двойное бронирование одного и того же слота.

Это означает:

- проверка доступности слота должна происходить внутри транзакции или эквивалентной гарантии
- на уровне базы должны быть ограничения или стратегия, не позволяющая двум конкурентным запросам создать пересекающиеся бронирования
- ошибка конфликта должна возвращаться как `409`, если слот уже занят

### 6. Ошибки должны быть доменными и предсказуемыми

Backend должен явно различать:

- `400` для невалидного payload или невозможного слота
- `404` для отсутствующего event type или override
- `409` для конфликтов состояния, например при двойном бронировании

Не нужно прокидывать сырой текст ошибок ORM или базы наружу в API.

Это не жёсткий каркас, но направление должно оставаться таким:

- `modules` для Nest composition
- `domain` для бизнес-логики
- `infrastructure` для БД и внешних деталей
- `common` для общих backend-механизмов

## Правила модулей

Backend должен быть разбит на небольшие модули по сценариям, а не по абстрактным техническим слоям.

Рекомендуемые модули:

- `health`
- `public-event-types`
- `public-slots`
- `public-bookings`
- `admin-event-types`
- `admin-availability`
- `admin-overrides`
- `admin-bookings`

Каждый модуль должен владеть:

- своим controller
- своим service
- своими зависимостями на repositories или domain services

## Правила контроллеров

Контроллеры должны:

- принимать входные данные
- вызывать application/domain service
- маппить результат в контрактный response shape
- не выполнять доменные вычисления

Контроллеры не должны:

- вычислять слоты
- решать, доступен ли интервал
- работать с транзакциями напрямую
- строить SQL-запросы

## Правила сервисов

Сервисы должны содержать основную прикладную и доменную логику.

Примеры обязанностей сервисов:

- вычислить публичные слоты для event type
- проверить, что слот попадает в 14-дневное окно
- проверить, что слот соответствует availability и overrides
- проверить отсутствие пересечений
- создать бронирование в конкурентно-безопасном режиме

Сервисы должны быть предсказуемыми и максимально независимыми от Nest specifics.

## Правила репозиториев

Репозитории отвечают только за доступ к данным.

Они должны:

- читать и писать event types
- читать и писать availability configuration
- читать и писать overrides
- читать и писать bookings
- поддерживать транзакционные операции, нужные для бронирования

Они не должны:

- определять, можно ли бронировать слот
- вычислять публичные слоты как бизнес-решение
- принимать решения об HTTP-ошибках

## Работа с контрактом и OpenAPI

Ожидаемый backend workflow:

1. обновить контракт в `../typespec/`
2. сгенерировать `openapi.json`
3. обновить backend-реализацию под новый контракт
4. прогнать contract checks и тесты

Если `Prism` используется в proxy/validation режиме вместе с backend, backend должен выдавать response, совместимый с контрактом без ручных обходов.

## Data mapping rules

Backend не должен возвращать ORM entities напрямую.

Нужно явно отделять:

- persistence model
- domain model
- API response model

Если структура почти совпадает, всё равно не нужно делать контроллер зависимым от ORM-объектов напрямую.

## Валидация на границе API

Входные данные должны валидироваться на границе API.

Но важно:

- синтаксическая и структурная валидация делается на границе
- доменная валидация делается в service-слое

Например:

- наличие `guestEmail` в payload проверяется на границе
- допустимость слота для бронирования проверяется в service

## Тестовые ожидания для backend
ТЕСТЫ НА ДАННОМ ЭТАПЕ НЕ ПИШУТСЯ

Backend должен покрываться тестами в первую очередь вокруг доменной логики.

Критический минимум:

- корректная генерация слотов
- исключение уже занятых интервалов
- отказ при бронировании вне окна 14 дней
- отказ при бронировании вне availability
- корректная работа overrides
- отказ при пересечении бронирований
- корректный `409` при конкурентном конфликте
- корректные `400`, `404`, `409` ответы на HTTP-слое

Рекомендуемый набор тестов:

- unit tests для slot generation и booking validation
- integration tests для repositories
- e2e tests для public и admin endpoints

## Ожидания от агентов в server/

При внесении изменений в backend:

- сохранять `TypeSpec-first` подход
- не дублировать типы без необходимости
- не переносить доменные правила в controllers
- держать время и timezone-логику централизованной
- явно разделять controller, service и repository responsibilities
- не возвращать ORM entities напрямую
- учитывать конкурентность при создании бронирований
- добавлять тесты для нетривиальной логики

## Вне области backend MVP

Следующее намеренно не входит в backend MVP:

- аутентификация и регистрация
- сложная ролевая модель и ACL
- мульти-owner архитектура
- интеграции с календарями
- нотификации
- платежи
- повторяющиеся бронирования
- фоновые очереди без реальной необходимости
- CQRS, event sourcing и другая архитектурная тяжёлая артиллерия без практической причины

Backend должен оставаться простым, модульным, предсказуемым и полностью подчинённым доменным правилам бронирования.
